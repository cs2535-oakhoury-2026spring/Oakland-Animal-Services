# ADR-011: Authentication - JWT with Refresh Token Rotation

## Context

The application requires secure authentication for multiple user types:
- **Staff** — Full access to all animal data and notes
- **Admin** — Staff permissions plus user management and activity logs
- **Volunteer** — Limited access with time-based account expiration
- **Device** — Automated systems that don't require password changes

Requirements include:
- Persistent login across browser refreshes
- Secure token storage
- Support for volunteer account expiration dates
- Password change enforcement
- Activity logging for security auditing
- Protection against token theft

Traditional session-based authentication was considered but rejected due to scalability concerns and the need for stateless backend architecture.

## Decision

Implement JWT-based authentication with refresh token rotation:

### Token Types

**Access Token** (short-lived, 15 minutes):
- Stored in `sessionStorage` (cleared when tab closes)
- Contains: `userId`, `username`, `role`, `expiresAt`, `mustChangePassword`
- Sent in `Authorization: Bearer <token>` header
- Used for all authenticated API requests

**Refresh Token** (long-lived, 7 days):
- Stored in httpOnly cookie (not accessible to JavaScript)
- Used only to obtain new access tokens
- Rotated on each refresh (single-use tokens)
- Tracked in DynamoDB `RefreshTokens` table

### Authentication Flow

1. **Login**: User provides username/password → Backend returns access token + sets refresh token cookie
2. **API Requests**: Frontend sends access token in header → Backend validates JWT
3. **Token Refresh**: When access token expires → Frontend calls `/auth/refresh` → Backend validates refresh token, issues new access + refresh tokens, invalidates old refresh token
4. **Logout**: Frontend calls `/auth/logout` → Backend deletes refresh token from database

### Role-Based Access Control

```js
// Middleware checks role from JWT payload
if (user.role === 'volunteer' && new Date(user.expiresAt) < new Date()) {
  return res.status(403).json({ message: 'Volunteer account has expired' });
}

if (['admin', 'staff'].includes(user.role)) {
  // Allow access
} else {
  return res.status(403).json({ message: 'Insufficient permissions' });
}
```

### Frontend Implementation

```jsx
// On app load, attempt silent refresh
useEffect(() => {
  const stored = sessionStorage.getItem("oas_token");
  const payload = stored ? decodeJwt(stored) : null;
  const isExpired = payload ? payload.exp * 1000 < Date.now() : true;

  if (stored && !isExpired) {
    applyToken(stored);
    api.refreshToken().catch(() => {
      // Keep current session if access token still valid
    });
  } else {
    api.refreshToken()
      .then(applyToken)
      .catch(() => {
        clearToken();
        setAccessToken(null);
      });
  }
}, []);
```

### Password Change Enforcement

JWT payload includes `mustChangePassword: true` for:
- Newly created accounts
- Admin-forced password resets

Frontend redirects to `ForcePasswordChangeScreen` if flag is set.

## Consequences

**Pros:**

- **Stateless backend** — No server-side session storage needed, horizontally scalable
- **Industry standard** — JWTs are well-understood and widely supported
- **Granular permissions** — Role embedded in token, no database lookup per request
- **Secure storage** — Refresh tokens in httpOnly cookies prevent XSS theft
- **Token rotation** — Refresh token rotation limits damage from token theft
- **Volunteer expiration** — Easy to enforce time-based access via `expiresAt` field
- **Cross-device logout** — Deleting refresh token forces re-authentication

**Cons:**

- **More complex than session cookies** — Requires managing two token types
- **Token size** — JWTs are larger than session IDs (~200 bytes vs ~20 bytes)
- **Cannot revoke access tokens** — Must wait for 15-minute expiry (refresh tokens can be revoked)
- **Clock skew issues** — Server and client clocks must be synchronized for `exp` validation
- **XSS vulnerability** — Access token in sessionStorage is vulnerable to XSS (mitigated by short lifetime)
- **CSRF protection needed** — Refresh endpoint requires CSRF token validation

## Alternatives Considered

**Session-based authentication** — Rejected because it requires server-side session storage (Redis/DynamoDB), doesn't scale horizontally as easily, and complicates load balancing.

**OAuth 2.0** — Rejected as overkill. OAuth is designed for third-party authorization, not first-party authentication.

**Single long-lived token** — Rejected for security. If stolen, would grant access until manual revocation. Short access tokens + refresh rotation limit exposure window.

**localStorage for tokens** — Rejected because localStorage persists across tab closures and is more vulnerable to XSS. sessionStorage + httpOnly cookies provide better security.

## Security Considerations

- Refresh tokens are single-use and invalidated after rotation
- Access tokens have short 15-minute expiry
- All tokens transmitted over HTTPS only
- CSRF protection on refresh endpoint
- Activity logging tracks all password changes and account modifications
- Volunteer accounts auto-expire based on `expiresAt` timestamp
