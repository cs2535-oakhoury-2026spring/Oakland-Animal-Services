# ADR-015: Data Fetching - Promise.all for Parallel Requests

## Context

When a user navigates to an animal's detail page (Portal component), three separate pieces of data are required:

1. **Pet data** — Basic info (name, species, breed, age, image, etc.)
2. **Medical notes** — Observer notes from veterinary staff
3. **Behavior notes** — Behavioral observations from handlers

Each data source requires a separate API call to different backend endpoints:
- `GET /api/pets/:petId`
- `GET /api/pets/:petId/observer-notes`
- `GET /api/pets/:petId/behavior-notes`

Initial implementation fetched data sequentially:
```jsx
useEffect(() => {
  const fetchData = async () => {
    const petData = await api.getPet(petId);        // ~300ms
    setPet(petData);
    const notesData = await api.getNotes(petId);     // ~300ms
    setNotes(notesData);
    const bNotes = await api.getBehaviorNotes(petId); // ~300ms
    setBehaviorNotes(bNotes);
  };
  fetchData();
}, [petId]);
```

**Total time: ~900ms** (three sequential network requests)

This created a poor user experience:
- Noticeable delay before any data appeared
- Pet card rendered before notes, creating layout shift
- Users perceived the app as slow

## Decision

Use `Promise.all()` to fetch all three data sources in parallel:

```jsx
useEffect(() => {
  const fetchData = async () => {
    setLoading(true);
    try {
      const [petData, notesData, bNotes] = await Promise.all([
        api.getPet(petId),
        api.getNotes(petId),
        api.getBehaviorNotes(petId)
      ]);
      
      setPet(petData);
      setNotes(notesData);
      setBehaviorNotes(bNotes);
    } catch (error) {
      console.error("Failed to fetch data:", error);
      // Individual API calls have their own fallbacks
    } finally {
      setLoading(false);
    }
  };
  
  fetchData();
}, [petId]);
```

**Total time: ~300ms** (all three requests start simultaneously, finish when slowest completes)

### Loading State

While data fetches, show skeleton placeholders for all three sections:
```jsx
if (loading) {
  return (
    <div>
      <Skeleton height={300} /> {/* Pet card */}
      <Skeleton height={200} /> {/* Notes section */}
      <Skeleton height={200} /> {/* Behavior section */}
    </div>
  );
}
```

This prevents layout shift and gives users immediate visual feedback.

## Consequences

**Pros:**

- **3x faster load time** — 300ms vs 900ms perceived performance improvement
- **Better UX** — All data appears simultaneously, no progressive loading jumps
- **Reduced perceived latency** — Skeleton loaders shown immediately
- **More efficient** — Backend can process requests in parallel too
- **Single loading state** — Simpler state management (one `loading` flag vs three)

**Cons:**

- **All-or-nothing** — If any request fails, entire Promise.all rejects (mitigated by individual API fallbacks)
- **Server load spike** — Three simultaneous requests per page load (acceptable at shelter scale)
- **Cannot show partial data** — Must wait for slowest request before rendering anything
- **Harder to debug** — Can't easily tell which specific request failed without additional error handling

## Error Handling Strategy

Each API function has its own fallback:
```js
// api.js
getPet: async (petId) => {
  try {
    const res = await fetch(`/api/pets/${petId}`);
    if (!res.ok) throw new Error("Failed to fetch pet");
    return await res.json();
  } catch (err) {
    console.warn("getPet: falling back to mock data", err);
    return mockPets.find(p => p.petId === String(petId)) || mockPets[0];
  }
}
```

This ensures that even if `Promise.all` rejects, individual functions still return usable data (either from API or mock fallback).

## Performance Metrics

**Before (sequential)**:
- First request starts: 0ms
- Second request starts: 300ms
- Third request starts: 600ms
- All data loaded: 900ms

**After (parallel)**:
- All requests start: 0ms
- All data loaded: 300ms (duration of slowest request)

**Real-world improvement**: 67% reduction in load time

## Alternatives Considered

**React Query / SWR** — Rejected for being overkill. These libraries add 10KB+ and provide caching/refetching features not needed for this use case. Data changes frequently enough that caching is not beneficial.

**Sequential with progressive rendering** — Rejected because it creates multiple layout shifts as each section loads, degrading UX.

**Lazy loading** — Rejected because all three data sources are above the fold and immediately needed.

**Promise.allSettled** — Considered but rejected because we want the loading state to wait for all requests. With `allSettled`, we'd need more complex logic to handle partial failures.

## Future Optimizations

If performance becomes an issue at scale:
- Implement backend endpoint that returns all three data sources in single request
- Add request deduplication (prevent multiple tabs from fetching same data)
- Consider React Query for automatic caching and background refetching
