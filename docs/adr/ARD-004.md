# ARD-004: Authentication and Authorization

## Context

Originally when talking to the Joe and Andy they were not too worried on the Authentication and Authorization part of the system. But we believe this is still an important aspect to consider. We want to make sure that this system is secure and robust enough so that there can be individuals thats logged in or has a organization based system for like ipads and devices used by multiple people inside of the building.

## Decision

### Authentication

Username and password authentication using the short-lived JWT + refresh token pattern.

- **Access token** — a JWT valid for 15 minutes, sent via `Authorization: Bearer <token>` header on every request.
- **Refresh token** — a long-lived token stored in the DB and sent to the client as an httpOnly cookie, valid for 30 days. Used to silently obtain a new access token when the current one expires. Users stay logged in indefinitely as long as they are active within the 30-day window, with no need to manually log in again.

On password reset or account deletion the refresh token is invalidated in the DB. The next time the client attempts a silent refresh it receives a 401 and the user is redirected to the login page.

If a volunteer account's `expiresAt` date has passed, the refresh endpoint rejects the request with a 401 even if the refresh token is otherwise valid.

### Roles

There are four roles:

- **admin** — credentials stored in `.env` (`ADMIN_USER` / `ADMIN_PASS`) and compared directly on login. Never stored in the database. Cannot be created through the UI.
- **staff** — created by admin. Can manage volunteers.
- **volunteer** — created by admin or staff. Always has an expiry date (`expiresAt`). When expired, access is revoked.
- **device** — created by admin only. Represents a shared device (e.g. `DogIpad1`). No expiry. Same API permissions as volunteer.

### Device Accounts

Shared devices (iPads etc.) get their own account. When a user submits a note on a shared device they add their name in a field, which appears in the note as `Jane Doe (DogIpad1)`. On a personal login it shows as just `Jane Doe`. This avoids requiring staff to log in and out on shared devices.

### Permission Matrix

| Action | Volunteer / Device | Staff | Admin |
|--------|---|---|---|
| All GETs, POST notes, PUT compatibility | ✅ | ✅ | ✅ |
| Create volunteer (with expiry) | ❌ | ✅ | ✅ |
| Edit / delete volunteers | ❌ | ✅ | ✅ |
| Create staff / device / admin | ❌ | ❌ | ✅ |
| Edit / delete staff / device / admin | ❌ | ❌ | ✅ |

### Middleware

Three middleware functions handle authorization:

- `authenticate` — verifies JWT and checks volunteer expiry. Required on all routes.
- `requireStaff` — allows `staff` and `admin` only.
- `requireAdmin` — allows `admin` only.

### Password Management

When an account is created or a password is reset, the admin or staff sets a temporary password and communicates it to the user. On next login the user is forced to change it before accessing the system. This ensures staff never knows a user's real password.

Staff can reset a volunteer's password (sets a new temp password and flags `mustChangePassword`). Only admin can reset staff, device, or admin passwords.

### User Record (DynamoDB)

```ts
{
  userId: string              // PK
  username: string            // unique
  passwordHash: string        // bcrypt hash
  role: "admin" | "staff" | "volunteer" | "device"
  deviceName?: string         // only for device accounts (e.g. "DogIpad1")
  expiresAt?: string          // only for volunteer accounts, ISO date string
  mustChangePassword: boolean // true when account is created or password is reset
  createdAt: string
}
```

### Refresh Token Record (DynamoDB)

Stored in a separate table so a user can have multiple active sessions (e.g. phone + desktop).

```ts
{
  tokenId: string    // PK, a random UUID
  userId: string     // FK to user
  expiresAt: string  // 30 days from creation, ISO date string
  createdAt: string
}
```

On password reset or account deletion all refresh tokens for that user are deleted.
