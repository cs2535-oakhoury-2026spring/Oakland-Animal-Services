# ADR-010: State Management - No Redux or Context API

## Context

The application needs to manage several types of state across multiple screens:
- User authentication (access token, user profile, role, expiry)
- Pet data (current pet, list of animals at location)
- Notes (medical observations, behavior notes, search results)
- UI state (dark mode, active tab, modals, loading states)

Common solutions for React state management include:
- **Redux** — Centralized global store with actions and reducers
- **Context API** — React's built-in global state solution
- **MobX, Zustand, Jotai** — Third-party state libraries
- **Local state with prop drilling** — Component-level state passed via props

The application has moderate complexity with ~15 screens and ~30 components.

## Decision

Use local component state with `useState` and prop drilling. Centralize API calls in `api.js` but keep state management at the component level. No global state library is used.

**Implementation pattern:**

```jsx
// App.jsx - Root component owns auth state
const [accessToken, setAccessToken] = useState(null);
const [currentUser, setCurrentUser] = useState(null);

// Pass state down to child components
<Portal 
  user={currentUser} 
  token={accessToken} 
  onLogout={handleLogout} 
/>

// Portal.jsx - Owns pet-specific state
const [pet, setPet] = useState(null);
const [notes, setNotes] = useState([]);
const [behaviorNotes, setBehaviorNotes] = useState([]);
```

**API layer separation:**

```js
// api.js - Centralized API calls
export const api = {
  getPet: async (petId) => { /* ... */ },
  getNotes: async (petId) => { /* ... */ },
  createNote: async (data) => { /* ... */ }
};
```

## Consequences

**Pros:**

- **Simplicity** — No boilerplate, reducers, actions, or store configuration
- **Easier to understand** — State flow is explicit through props
- **No external dependencies** — Smaller bundle size, one less library to learn
- **Faster development** — No need to wire up Redux DevTools or write action creators
- **Easier debugging** — Can trace state changes by following props
- **Good for small teams** — Less architectural overhead for 2-3 developers

**Cons:**

- **Prop drilling** — Some props pass through 3-4 component levels
- **State duplication** — Some data (like `user`) exists in multiple components
- **No time-travel debugging** — Can't replay actions like with Redux DevTools
- **Harder to scale** — Would need refactoring if app grows to 100+ components
- **No centralized state logic** — Similar state updates repeated across components
- **Performance** — Re-renders can cascade through prop chains (though not a problem at current scale)

**Alternatives Considered:**

- **Redux** — Rejected as overkill for current app size. Would add ~20KB to bundle and significant boilerplate.
- **Context API** — Considered but rejected because most state is screen-specific, not truly global. Auth state is the only candidate, but `sessionStorage` handles persistence.
- **Zustand** — Lightweight alternative to Redux, but still adds dependency and learning curve for minimal benefit at current scale.

**Migration Path:**

If the app grows beyond ~50 components or state management becomes painful, migration to Context API or Zustand would be straightforward since state is already centralized in parent components.

**Notes:**

The decision to avoid global state management is intentional and appropriate for the current application size. The principle of "use the simplest solution that works" applies here.
