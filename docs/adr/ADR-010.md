# ADR-010: Local State with Prop Drilling over Global State Management

## Context

After decomposing the single-file frontend into 40+ components (ADR-009), state needed to flow between components. The standard React ecosystem offers several approaches:

1. **Redux** -- Global store with actions/reducers. Adds ~10KB and significant boilerplate.
2. **React Context API** -- Built-in provider/consumer pattern for shared state.
3. **Local state with prop drilling** -- Each component owns its state; parent passes data to children via props.

The application has a relatively flat component tree: `App.jsx` handles auth and routing, screens render directly under it, and components nest 1-2 levels deep at most.

## Decision

Use local `useState`/`useEffect` hooks with prop drilling. No global state library or Context providers.

- `App.jsx` owns authentication state (`currentUser`, `accessToken`, `darkMode`) and passes it to screens.
- Each screen owns its own data-fetching state (e.g. `HomeScreen` fetches and owns `animals[]`).
- Shared behavior is extracted into custom hooks (`useResponsive`, `useFocusTrap`, `useEscapeKey`) in `hooks.js`.
- API calls are centralized in `api.js` as a plain module with a module-level auth token -- no React state involved.

## Consequences

**Pros:**

1. Zero additional dependencies -- no Redux, MobX, or Zustand to install or learn
2. State is co-located with the component that uses it, making data flow easy to trace
3. No boilerplate -- no action creators, reducers, selectors, or providers
4. Custom hooks (`useResponsive`, `useFocusTrap`) share behavior without sharing state

**Cons:**

1. Props must be threaded through intermediate components that don't use them (e.g. `darkMode`, `onLogout` passed through every screen)
2. If the component tree deepens significantly, prop drilling becomes unwieldy
3. No built-in mechanism for cross-component communication without lifting state up to a common ancestor
