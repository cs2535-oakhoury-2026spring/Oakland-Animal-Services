# ADR-002: Architecture For Filtering Notes

## Context

OAS has a problem where volunteers would insert duplicated notes for a pet. This would cause frustration with medical staff seeing "dogA has a limp" reported x5.

## Decision

When a volunteer inputs a note, the content is checked at a debounced interval and run through a keyword extraction and fuzzy matching algorithm to find similar existing notes for that pet.

The search logic was initially implemented server-side (`/api/search`) using the `compromise` NLP library for verb/noun normalization and `fastest-levenshtein` for fuzzy matching. Following a review of bundle size and performance, the decision was made to move the logic entirely to the client side.

**Current implementation:**

- Search runs in the browser using notes already fetched for the pet view — no extra network requests per keystroke
- `compromise` and `fastest-levenshtein` were removed as dependencies entirely
- Verb stemming (`-ing`/`-ed`) and noun singularization (`-ies`, `-ves`, `-s`, etc.) are handled with lightweight manual rules
- Levenshtein distance is computed using a port of [js-levenshtein](https://github.com/gustf/js-levenshtein) (MIT) in `frontend/src/utils/levenshtein.js`
- The search logic is edited in `src/utils/observerNoteSearch.ts`. After changes, compile the TypeScript and copy the build output to `frontend/src/utils/noteSearcher.js`
- Keeping the logic in `.ts` preserves type safety and makes it easier to maintain. The compiled `.js` copy is intentional to avoid adding a TypeScript build step to the frontend while still keeping the actual development in TypeScript
- Preprocessed note tokens are cached in a module-level `Map` for the browser session, keyed by `author|timestamp|title|content`, so notes are only tokenized once per page load
- The `/api/search` server endpoint has been preserved but is no longer called by the frontend

## Consequences

**Pros:**

- Zero server load per search — all matching happens in the browser
- No network latency per keystroke
- Removing `compromise` (~230KB zipped) kept the frontend bundle at ~64KB
- Cache makes repeated searches near-instant (0.0ms preprocess after first search) and is reset on page reload.

**Cons:**

- All notes for the pet must be loaded before search can run (already the case in the current architecture)
- Irregular verb/noun forms (e.g. "ran" → "run", "mice" → "mouse") are not normalized — manual rules only handle common patterns
- Search logic is duplicated between `.ts` (server) and `.js` (client) and must be kept in sync manually
