# ADR-009: Styling Strategy - Inline Styles to CSS Variables

## Context

The original frontend implementation used inline JavaScript style objects for all styling:

```jsx
<div style={{ 
  backgroundColor: darkMode ? "#1a1a1a" : "#f2f2f2",
  color: darkMode ? "#f0f0f0" : "#1a1a1a",
  padding: 16,
  borderRadius: 10 
}}>
```

This approach was chosen to enable dynamic theming where colors changed programmatically based on the `darkMode` state. A centralized `themes` object stored all colors:

```jsx
const themes = {
  light: { bg: "#f2f2f2", textPrimary: "#1a1a1a", ... },
  dark: { bg: "#1a1a1a", textPrimary: "#f0f0f0", ... }
};
const c = darkMode ? themes.dark : themes.light;
```

While this worked, it became verbose and harder to maintain as the application grew. Each component needed to receive theme colors through props, and style objects cluttered the JSX.

## Decision

Migrate to CSS files with CSS custom properties (CSS variables) for theming:

1. **CSS Variables for Colors**: Define all theme colors as CSS custom properties in `styles/variables.css`
2. **data-theme Attribute**: Toggle between themes by setting `data-theme="light"` or `data-theme="dark"` on the `<html>` element
3. **Separate CSS Files**: Each component gets a `.css` file with standard CSS syntax
4. **localStorage Persistence**: Store theme preference in `localStorage` and apply on page load

**Implementation:**

```css
/* variables.css */
:root[data-theme="light"] {
  --clr-bg: #f2f2f2;
  --clr-text-primary: #1a1a1a;
  --clr-card-bg: #ffffff;
  /* ... */
}

:root[data-theme="dark"] {
  --clr-bg: #1a1a1a;
  --clr-text-primary: #f0f0f0;
  --clr-card-bg: #2a2a2a;
  /* ... */
}
```

```css
/* Portal.css */
.portal-container {
  background-color: var(--clr-bg);
  color: var(--clr-text-primary);
  padding: 16px;
  border-radius: 10px;
}
```

```jsx
// App.jsx
const toggleDarkMode = (val) => {
  localStorage.setItem("darkMode", val);
  document.documentElement.setAttribute("data-theme", val ? "dark" : "light");
  setDarkMode(val);
};
```

## Consequences

**Pros:**

- **Standard CSS syntax** — Easier for designers and developers familiar with CSS
- **Better performance** — Styles are parsed once by browser, not recalculated on every render
- **Smaller bundle size** — No JavaScript theme objects duplicated across components
- **Cleaner JSX** — No cluttered inline style objects
- **Browser caching** — CSS files can be cached independently
- **Better tooling** — CSS linting, autocomplete, and syntax highlighting work properly
- **Instant theme switching** — CSS variables update immediately when `data-theme` changes

**Cons:**

- **Browser support** — CSS custom properties don't work in IE11 (not a concern for this project)
- **Less dynamic** — Can't compute styles as easily as JavaScript (e.g., conditional logic requires class names)
- **All colors must be defined upfront** — Can't dynamically generate colors without additional JavaScript
- **Migration effort** — Required converting 2,500+ lines of inline styles to CSS

**Notes:**

The migration maintained all existing functionality including dark mode toggling. The `data-theme` attribute approach is a modern best practice for implementing theme systems and is more maintainable than inline JavaScript styles for applications of this size.
