# ADR-009: Migrating from Single-File to Multi-Component Architecture

## Context

The original frontend was a single `App.jsx` file containing all UI logic -- login, home screen, animal portal, note creation modals, user management, activity logs, and more. At its peak the file exceeded 4,000 lines.

Problems with the single-file approach:

1. **Merge conflicts** -- Multiple developers editing the same file caused constant Git conflicts.
2. **Readability** -- Finding a specific feature required scrolling through thousands of lines; no clear boundaries between concerns.
3. **Performance** -- Every state change re-evaluated the entire render tree since all components shared a single scope.
4. **Testing** -- Impossible to unit-test individual features in isolation.

## Decision

Decompose the monolith into a component-per-file architecture organized by role:

```
frontend/src/
  App.jsx                        # Root: auth, routing, dark mode
  api.js                         # All API calls
  hooks.js                       # Shared hooks (useResponsive, useFocusTrap, useEscapeKey)
  constants.js                   # Shared constants
  utils.js                       # Shared utilities
  styles/
    variables.css                # Theme CSS custom properties
    global.css                   # Shared classes and animations
  screens/
    LoginScreen.jsx / .css
    HomeScreen.jsx / .css
    ActivityLogScreen.jsx / .css
    LocationsPage.jsx / .css
    UserManagement/
      UserManagementScreen.jsx / .css
      CreateUserModal.jsx
      ResetPasswordModal.jsx
      BatchImportModal.jsx
      EditExpiryModal.jsx
  components/
    Portal.jsx / .css            # Animal detail view
    DesktopPortal.jsx / .css     # Desktop-optimized layout
    AnimalSelection.jsx / .css
    UserDropdown.jsx / .css
    notes/
      MedicalNoteCard.jsx
      BehaviorNoteCard.jsx
      CreateNoteModal.jsx
      CreateBehaviorNoteModal.jsx
      EditNoteModal.jsx
      EditBehaviorNoteModal.jsx
      SimilarNotesPreview.jsx
```

**Separation rules applied:**

- **Screens** are top-level views tied to URL state (`view=activity`, `view=users`, `view=locations`).
- **Components** are reusable UI pieces rendered within screens.
- **Notes** are grouped in a `notes/` subfolder since they share data structures and patterns.
- Each `.jsx` file has a co-located `.css` file for component-specific styles.
- Shared logic (API calls, hooks, constants) lives in top-level utility files.

`App.jsx` now only handles authentication state, URL parameter routing, and dark mode -- roughly 280 lines.

## Consequences

**Pros:**

1. Merge conflicts dropped significantly -- developers work on separate files
2. Each component is self-contained and independently readable
3. Co-located CSS keeps styles next to the components they affect
4. Easier onboarding -- new developers can understand one component at a time
5. Logical grouping (screens vs components vs notes) mirrors the app's navigation structure

**Cons:**

1. More files to navigate (40+ JSX/CSS files vs 1 monolith)
2. Props must be passed through multiple levels (no global state manager) -- addressed in ADR-010
3. Migration required careful extraction to avoid breaking shared state
4. Import paths are longer and must be maintained when files move
