# ADR-007: Activity Logging

## Context

As the shelter system grows, staff and admins need visibility into what is happening, who added or deleted notes, when accounts were created or removed, and when passwords were changed. Without a log, there is no way to audit past activity or investigate issues after the fact.

## Decision

### What Is Logged

Three categories of events are captured:

- **behaviorNote** — note created, deleted, or bulk-deleted by pet
- **observerNote** — note created, deleted, bulk-deleted, or status changed
- **authEvent** — password changed, user account created, updated, or deleted

Routine events (logins, logouts, token refreshes) are intentionally not logged to keep the feed readable. Only meaningful changes are recorded.

### Log Record

Each entry is flat and simple:

```ts
{
  logId: string       // UUID, primary key
  date: string        // YYYY-MM-DD, used for partitioning
  timestamp: string   // ISO string, used for ordering
  tag: "behaviorNote" | "observerNote" | "authEvent"
  actor: string       // username of whoever performed the action
  action: string      // e.g. CREATED, DELETED, STATUS_CHANGED, PASSWORD_CHANGED
  jsonData?: object   // event-specific detail (petId, noteId, role, etc.)
}
```

### DynamoDB Table Design

The table uses a single constant partition key (`pk = "ALL"`) with a composite sort key (`sk = timestamp#logId`). This gives naturally chronological ordering on every query with no extra work, which is the primary access pattern.

This is a deliberate tradeoff: a single partition becomes a hot spot at very high write volume. For a shelter-scale system this is not a concern — the write rate will never be high enough to hit DynamoDB throughput limits.

### Access Control

- **Staff** — can view `behaviorNote` and `observerNote` logs only
- **Admin** — can view all tags including `authEvent`

Staff requesting the `authEvent` tag receive a 403. If no tag filter is specified, staff default to note tags only.

### Querying

`GET /api/activity` supports:
- `tags` — comma-separated tag filter
- `actor` — filter by username
- `action` — filter by action name
- `from` / `to` — ISO date range on timestamp
- `limit` / `page` — pagination (default limit 20, page 1)

Results are returned newest first.

### Write Strategy

`logActivity()` is a fire-and-forget utility. Logging failures are printed to stderr but never surface as errors to the caller. This ensures a DynamoDB hiccup cannot break a note submission or user management action.

### Automatic Table Creation

The `createActivityLogTable` script is imported and called on server startup. Since it checks for table existence before creating, it is safe to run on every boot. This means no manual setup step is required after deploy.

## Alternatives Considered

**Separate tables per tag** — rejected for complexity. A single table with a `tag` field and `FilterExpression` is simpler and sufficient for this volume.

**Logging logins** — rejected for now. Login events would dominate the feed and make it harder to spot meaningful changes. Can be added later if needed.
