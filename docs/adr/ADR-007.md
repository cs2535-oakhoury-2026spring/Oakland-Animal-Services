# ADR: Logging And Activity Audit Trail

## Context

Staff need to understand what actions were taken on the system — who created a note, when was a pet record updated, who changed a user's role. This is critical for accountability, debugging, and compliance.

Without an audit trail, staff cannot answer questions like "who deleted this note?" or "when was this change made?" This creates liability and makes troubleshooting difficult.

The system must log user actions to a durable store (database, not just console logs which are ephemeral).

## Decision

Implement an activity audit log stored in DynamoDB table `ActivityLog` with entries:

```typescript
{
  id: string,
  timestamp: Date,
  actor: string,           // username who performed action
  action: string,          // e.g., "NOTE_CREATED", "USER_PASSWORD_CHANGED"
  resourceType: string,    // e.g., "pet", "note", "user"
  resourceId: string,      // e.g., petId,
  tag: string,            // category: "note", "user", "auth"
  jsonData: {},           // additional context (what changed, old value, etc)
}
```

Log user actions via `logActivity()` utility called from controllers/services:

```typescript
logActivity({
  tag: "note",
  actor: req.user.username,
  action: "NOTE_CREATED",
  jsonData: { petId, noteContent: content.substring(0, 100) },
});
```

Staff can query the activity log via admin dashboard to see who did what when. Staff can only see note related logs for pets while admins can see all logs.

## Consequences

### Benefits

1. **Accountability** — Staff can audit who performed each action; discourages accidental/malicious changes
2. **Security incident response** — If account is compromised, audit log shows what was accessed/modified

### Drawbacks

1. **Storage overhead** — Every action writes to database; activity log table grows quickly 
2. **Performance impact** — Logging happens synchronously; adds latency to every request
3. **Privacy concerns** — Logging user actions may require explicit consent or privacy policy updates
