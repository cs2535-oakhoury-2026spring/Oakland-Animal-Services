# ADR-012: Responsive Design - Breakpoint Strategy

## Context

The application is used in two distinct environments:

1. **Office staff** — Desktop computers with large monitors (1920x1080+) for data entry and reviewing animal records
2. **Field volunteers** — Mobile devices (phones/tablets) for scanning QR codes at kennels and adding quick notes

These use cases require different UI layouts:
- Desktop users benefit from seeing the pet card and notes side-by-side
- Mobile users need a single-column layout optimized for one-handed use
- Tablet users (768px - 1024px) could potentially use either layout

Common responsive design approaches include:
- Multiple breakpoints (320px, 768px, 1024px, 1440px)
- Fluid layouts with `clamp()` and responsive units
- CSS Grid with `auto-fit` / `auto-fill`
- Mobile-first progressive enhancement

## Decision

Use a **single breakpoint at 768px** with two distinct layouts:

### Below 768px — Mobile Layout
- Single-column stacked layout
- Pet card at top with collapsible "More Details" section
- Tabs below (Summary, Medical, Behavior)
- Full-width tab content
- Touch-friendly buttons (44px minimum height)

### Above 768px — Desktop Layout
- Two-column grid layout
- Fixed pet card on left (400px wide)
- Scrollable tab content on right
- Larger font sizes and spacing
- Mouse-optimized interactions

### Implementation

**JavaScript-based detection**:
```jsx
// hooks.js
export function useResponsive() {
  const [width, setWidth] = useState(window.innerWidth);
  
  useEffect(() => {
    const handleResize = () => setWidth(window.innerWidth);
    window.addEventListener("resize", handleResize);
    return () => window.removeEventListener("resize", handleResize);
  }, []);
  
  return {
    width,
    isPhone: width <= 480,
    isTablet: width > 480 && width < 768,
    isDesktop: width >= 768
  };
}
```

**Component routing**:
```jsx
// Portal.jsx
const r = useResponsive();

if (r.width >= 768) {
  return <DesktopPortal {...props} />;
}

return (
  // Mobile layout
  <div className="portal-mobile">
    {/* Single column layout */}
  </div>
);
```

## Consequences

**Pros:**

- **Simple binary decision** — Either mobile or desktop, no intermediate states
- **Clear UX** — Each mode optimized for its environment
- **Easy to maintain** — Two distinct components instead of complex conditional rendering
- **Predictable behavior** — No guessing what layout users will see
- **Industry standard** — 768px is widely recognized as tablet breakpoint
- **Better testing** — Can test two specific layouts instead of many permutations

**Cons:**

- **No optimized tablet view** — Tablets (768-1024px) use desktop layout, may have wasted space
- **Sharp transition** — Resizing window crosses breakpoint causes layout jump
- **Duplicate component code** — `DesktopPortal` and mobile `Portal` share logic but render separately
- **No fluid scaling** — Font sizes and spacing are fixed for each mode
- **Landscape phone ambiguity** — Large phones in landscape (~800px width) get desktop layout

## Alternatives Considered

**Multiple breakpoints (320, 768, 1024, 1440)** — Rejected for complexity. Would require 4 different layouts with minimal UX benefit. Most users are clearly mobile (phone) or desktop (office computer).

**Fluid responsive layout** — Rejected because the two use cases are fundamentally different. Mobile needs stacked, desktop needs side-by-side. No graceful "in-between" exists.

**CSS Grid auto-fit** — Rejected because grid can't handle the different interaction patterns (touch vs mouse) and information density requirements.

**Container queries** — Considered for future adoption once browser support improves, but not available in target browsers (2024).

## Notes

The 768px breakpoint aligns with:
- iPad portrait mode (768 x 1024)
- Industry standard for "tablet" breakpoint
- Bootstrap's `md` breakpoint
- Most design systems' mobile/desktop cutoff

This decision can be revisited if analytics show significant tablet traffic (768-1024px) with poor UX.
