# ADR-008: Migrating from Inline Styles to CSS Variables and External Stylesheets

## Context

The initial frontend implementation used JavaScript inline styles for all component styling. Every component defined style objects directly in the render logic:

```jsx
const cardStyle = {
  backgroundColor: darkMode ? "#2a2a2a" : "#ffffff",
  border: `1px solid ${darkMode ? "#3a3a3a" : "#cccccc"}`,
  borderRadius: 12,
  boxShadow: "0 2px 8px rgba(0,0,0,0.06)",
};
```

This caused several problems:

1. **Dark mode duplication** -- Every component had ternary expressions for light/dark colors, duplicated across 20+ components.
2. **No hover/focus/animation support** -- Inline styles cannot use pseudo-selectors (`:hover`, `:focus`) or `@keyframes`, so interactive states required `onMouseEnter`/`onMouseLeave` handlers in JS.
3. **Large component files** -- Style objects added 50-150 lines of non-functional code to each component file, making logic harder to follow.
4. **Inconsistent theming** -- Colors were hardcoded per-component with no single source of truth, leading to slight color mismatches across screens.

## Decision

Migrate to CSS custom properties (variables) with external `.css` files per component.

**Theme variables** are defined once in `frontend/src/styles/variables.css`:

```css
:root {
  --clr-bg: #f2f2f2;
  --clr-card-bg: #ffffff;
  --clr-text-primary: #1a1a1a;
  /* ... */
}

[data-theme="dark"] {
  --clr-bg: #1a1a1a;
  --clr-card-bg: #2a2a2a;
  --clr-text-primary: #f0f0f0;
  /* ... */
}
```

**Global reusable classes** are in `frontend/src/styles/global.css` (e.g. `.oas-card`, `.oas-btn-primary`, `.oas-modal-overlay`).

**Component-specific styles** live in co-located `.css` files (e.g. `Portal.css`, `HomeScreen.css`).

Dark mode toggling sets a `data-theme` attribute on `<html>`, and all CSS variables swap automatically without any JS ternaries.

## Consequences

**Pros:**

1. Dark mode is handled in one file (`variables.css`) instead of duplicated across every component
2. Hover, focus, and animation states use native CSS pseudo-selectors
3. Component files are shorter and focused on logic
4. Consistent theming via shared variables -- changing a color in one place updates the entire app

**Cons:**

1. CSS class naming must be managed carefully to avoid collisions (mitigated by component-scoped file naming)
2. Migration required touching every component file to remove inline style objects
3. Developers must maintain both `.jsx` and `.css` files per component
