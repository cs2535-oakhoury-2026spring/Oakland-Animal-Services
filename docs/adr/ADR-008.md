# ADR-008: Frontend Architecture - Single File vs. Multi-Component

## Context

The frontend was initially built as a single 2,571-line `App.jsx` file with all components, styles, and logic in one place. This approach worked well for rapid prototyping and allowed for quick iteration during early development. However, as the application grew to include authentication, user management, activity logging, and more complex features, maintainability and team collaboration became concerns.

The single-file approach made it difficult to:
- Navigate to specific components quickly
- Work on features in parallel without merge conflicts
- Reuse components across different screens
- Test individual components in isolation
- Onboard new developers who needed to understand the entire codebase at once

## Decision

Refactor the frontend to a multi-file component architecture with the following structure:

```
frontend/src/
├── App.jsx                    # Root component (routing and auth logic only)
├── api.js                     # Centralized API layer
├── utils.js                   # Helper functions
├── hooks.js                   # Custom React hooks
├── constants.js               # Application constants
├── Icons.jsx                  # SVG icon components
├── components/                # Reusable UI components
│   ├── Portal.jsx + Portal.css
│   ├── AnimalSelection.jsx + AnimalSelection.css
│   ├── ErrorScreen.jsx + ErrorScreen.css
│   ├── notes/                 # Note-related components
│   │   ├── MedicalNoteCard.jsx
│   │   ├── BehaviorNoteCard.jsx
│   │   └── ...
│   └── ...
├── screens/                   # Full-page screens
│   ├── HomeScreen.jsx + HomeScreen.css
│   ├── LoginScreen.jsx + LoginScreen.css
│   ├── ActivityLogScreen.jsx
│   ├── UserManagement/
│   └── ...
└── styles/
    ├── global.css             # Global styles and resets
    └── variables.css          # CSS custom properties
```

Each component gets its own `.jsx` file with a corresponding `.css` file for styles. Related components are grouped into subdirectories (e.g., `notes/`, `UserManagement/`).

## Consequences

**Pros:**

- **Better organization** — Components are easy to locate by name or functionality
- **Team collaboration** — Multiple developers can work on different components without conflicts
- **Code reusability** — Components can be imported and reused across screens
- **Easier testing** — Individual components can be tested in isolation
- **Clearer responsibility** — Each file has a single, well-defined purpose
- **Better IDE support** — Autocomplete and navigation work better with smaller files
- **Reduced cognitive load** — Developers only need to understand one component at a time

**Cons:**

- **More files to manage** — 50+ component files vs. 1 file
- **Import overhead** — Need to manage import statements across files
- **Initial migration cost** — Time spent refactoring existing code
- **Navigation complexity** — Need to jump between files to trace data flow
- **Build complexity** — Slightly more complex module resolution

**Notes:**

The refactoring preserved all existing functionality while improving long-term maintainability. The App.jsx file was reduced from 2,571 lines to 280 lines, focusing only on routing and authentication logic.
